# Issues Analysis: ROI, Cost & Business Value Agents

> Audit date: 2026-04-02
> Scope: `roi_agent.py`, `cost_agent.py`, `business_value_agent.py`, `pricing.py`

---

## Summary

The three financial agents form a pipeline: **Cost → Business Value → ROI**. The ROI agent is purely computational (no LLM), consuming outputs from the other two. The core problem pattern is: **LLM-generated numbers flow through without math verification, wrapped in arbitrary constants and fuzzy key lookups, producing a dashboard that looks precise but has wide error bars that are invisible to the user.**

---

## Critical Issues (affect correctness of ROI headline)

### 1. ROI Formula Excludes Implementation Costs
**File:** [roi_agent.py](src/python-api/agents/roi_agent.py#L348)
**Line:** `roi_mid = ((val_mid - annual_cost) / annual_cost) * 100`

`annual_cost` is Azure run-rate only. Implementation costs (`impl_cost`, `change_cost`) are calculated but excluded from the headline ROI. A project costing $50K/yr in Azure + $200K in implementation shows the same ROI as one with zero implementation cost. The `year1_investment` field captures this correctly, but `roi_percent` (the headline) does not.

**Impact:** ROI is inflated, especially for high-touch implementations.

### 2. Payback Period Ignores Implementation Cost
**File:** [roi_agent.py](src/python-api/agents/roi_agent.py#L350)
**Line:** `payback_months = round((annual_cost * 12 / val_mid), 1)`

Uses only Azure annual cost, not Year 1 total investment. A $50K Azure + $200K implementation project shows ~2 month payback when it should be ~6 months.

**Impact:** Payback period is underestimated.

### 3. No Math Verification on LLM-Generated Value Range
**File:** [business_value_agent.py](src/python-api/agents/business_value_agent.py#L299-L310)

The `annual_impact_range` (low/high) comes directly from the LLM with zero validation:
- No check that `low ≤ high`
- No check that individual driver amounts sum to the range
- No sanity bound (e.g., value > 100× cost should flag)
- No check that the LLM actually did the arithmetic correctly

**Impact:** The entire value side of ROI is a single unverified LLM output.

### 4. Estimated Baseline is Arbitrary
**File:** [roi_agent.py](src/python-api/agents/roi_agent.py#L44)
**Constant:** `ESTIMATED_BASELINE_MULTIPLIER = 1.5`

When no current-state baseline is provided, current cost = 1.5× Azure cost. This is a made-up number. A $10K/mo Azure deployment gets a $15K/mo "current cost" — often wildly wrong in either direction for real enterprises.

**Impact:** When `is_estimated=True`, the monthly savings, cost comparison, and business case are all based on a fabricated baseline.

## High-Severity Issues (structural/reliability)

### 5. Fragile Regex Extraction of AI Coverage from BV Drivers
**File:** [roi_agent.py](src/python-api/agents/roi_agent.py#L256-L266)

`_extract_coverage_from_drivers()` parses BV driver metric strings like `"10–20% time savings"` via regex to determine how much to reduce labor costs in the future-state model. If the LLM formats the metric differently (e.g., `"saves approximately 15 percent"`), extraction fails silently and falls back to 25%.

**Impact:** Future-state cost model is unstable — small LLM prompt changes cause different output.

### 6. Fuzzy Key Resolution Cascade (`_resolve_sa`)
**File:** [roi_agent.py](src/python-api/agents/roi_agent.py#L65-L108)

Three-level matching (exact → substring → keyword) with no logging of which level matched. A key `current_annual_cloud_cost` would match the `spend` keyword path and return a value that might be the wrong semantic. This is a band-aid for the lack of a typed shared-assumptions schema.

**Impact:** Silent incorrect value resolution, impossible to debug.

### 7. `shared_assumptions` is Schema-Free
**File:** [state.py](src/python-api/agents/state.py#L32)

`shared_assumptions: dict[str, Any]` — keys are arbitrary strings generated by LLM prompts. The three agents each have their own lists of expected key names (`_CURRENT_SPEND_KEYS`, `_LABOR_RATE_KEYS`) and fuzzy match against them. No validation, no schema, no type enforcement.

**Impact:** Agents may read the wrong value or miss values entirely.

### 8. Hardcoded Future-State Reduction Factors
**File:** [roi_agent.py](src/python-api/agents/roi_agent.py#L207-L227)

- Overhead always reduced by 10% (`* 0.9`)
- Error/rework reduction defaults to 50% (`error_reduction` assumption or 50)
- "User-provided" baseline always split 60/40 labor/overhead
- These are arbitrary constants with no connection to the actual solution being proposed.

**Impact:** Future-state cost model doesn't reflect the actual architecture.

### 9. Sequential Pricing API Calls
**File:** [cost_agent.py](src/python-api/agents/cost_agent.py#L392)

There's a `TODO` to parallelize. Each Azure Retail Prices API call has a 15s timeout. 10 services = up to 150s of serial waiting. The comment says 10s but `pricing.py` uses `timeout=15`.

**Impact:** Slow user experience on every cost estimate.

### 10. Pricing Fallback to Median SKU
**File:** [pricing.py](src/python-api/services/pricing.py#L150-L152)

When no SKU match is found, `_find_best_match()` returns the median-priced item from all results for that service. A "B1" App Service lookup might return a "P3v3" price. No indication in the output that the matched SKU is unrelated.

**Impact:** Cost can be off by 10-50× for mismatched SKUs.

## Medium-Severity Issues

### 11. Risk Reduction is Always 3% of Current Spend
**File:** [roi_agent.py](src/python-api/agents/roi_agent.py#L293)
`risk_raw = round(current_annual * 0.03)`

Fixed 3% regardless of the actual risk profile. Not connected to any risk analysis. Excluded only when it's <5% of the other value (the materiality check is fine, but the 3% itself is arbitrary).

### 12. Forced "Exactly 3 Drivers" in Business Value
**File:** [business_value_agent.py](src/python-api/agents/business_value_agent.py#L183)

The prompt demands `"EXACTLY 3 value drivers"`. For simple use cases (e.g., "migrate a static website"), this forces the LLM to invent a third driver. For complex use cases, it forces collapsing multiple real drivers into 3.

### 13. OpenAI Pricing is Hardcoded and Likely Stale
**File:** [pricing.py](src/python-api/services/pricing.py#L82-L94)

`$0.006/request` assumes GPT-4o with 800 input + 400 output tokens. Token counts per request vary wildly by use case. GPT-4o pricing changes over time. No mechanism to update.

### 14. `_calculate_monthly` Default Fallthrough
**File:** [cost_agent.py](src/python-api/agents/cost_agent.py#L492)
`else: return unit_price * 730`

Unrecognized units silently get treated as hourly pricing (×730 hours/month). A per-GB-day unit would be wildly overcharged.

### 15. Multi-Region Overhead is Fixed 40%
**File:** [cost_agent.py](src/python-api/agents/cost_agent.py#L225)

`overhead = sum(...) * 0.4` for all services. Real multi-region overhead varies dramatically (active-active vs active-passive, data replication patterns, etc.).

### 16. Web Search Failure Degrades Silently
**File:** [business_value_agent.py](src/python-api/agents/business_value_agent.py#L161-L162)

When `search_industry_benchmarks` fails, confidence stays "moderate" and the prompt switches to a "be honest" mode. The user sees the same dashboard quality indicator whether benchmarks were found or not.

### 17. 3-Year Projection Uses Fixed Adoption Ramp
**File:** [roi_agent.py](src/python-api/agents/roi_agent.py#L46)

`ADOPTION_RAMP = [0.50, 0.85, 1.00]` — every scenario, regardless of complexity, org size, or change readiness, gets the same adoption curve. A simple tool migration and a full digital transformation share the same projection.

### 18. `productivityGains` is Always 0
**File:** [roi_agent.py](src/python-api/agents/roi_agent.py#L766)

`"productivityGains": 0` with comment "folded into hard_savings to avoid double-counting". The category exists in the dashboard/schema but is permanently zero — confusing for anyone reading the business case.

## Low-Severity / Code Quality Issues

### 19. No Caching of Pricing API Calls
Same service/SKU/region is re-queried on every run. No in-memory or session-level cache.

### 20. Silent JSON Parse Fallbacks in Cost + BV Agents
Both agents parse LLM JSON output with a try/except that falls through to generic fallback data. For cost_agent, the LLM service mapping failure creates "Standard" SKU placeholders that are then priced via the API — potentially returning valid but wrong prices.

### 21. Hard Savings Scale-Down Without Transparency
**File:** [roi_agent.py](src/python-api/agents/roi_agent.py#L285-L288)

When hard savings exceed current annual, they're scaled proportionally. The dashboard doesn't indicate this happened — the user sees reduced numbers with no explanation.

### 22. Assumption Value Types Are Stringly-Typed
Values in `shared_assumptions` and `user_assumptions` are cast with `float()` everywhere, suggesting they could arrive as strings, ints, or floats. No validation at the boundary.

### 23. Exception Swallowing Across All Agents
All three agents catch broad `Exception` and return fallback data. The user gets a dashboard with "moderate confidence" that may be based entirely on default/fallback numbers. No error propagation to the frontend.
