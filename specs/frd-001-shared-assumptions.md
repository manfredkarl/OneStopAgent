# FRD-001: Typed Shared Assumptions

**Feature ID**: F-001
**Status**: Draft
**Priority**: P0
**Last Updated**: 2026-04-02

## Description

The multi-agent pipeline (Cost → Business Value → ROI) shares baseline
assumptions via `AgentState.shared_assumptions`, a schema-free `dict[str, Any]`
whose keys are generated by an LLM. Each downstream agent independently
fuzzy-matches against its own private key list to resolve values like
`current_annual_spend` or `hourly_labor_rate`. The same raw value can resolve
differently in each agent, producing silently inconsistent financial models.

Additionally, the Cost and Business Value agents each LLM-generate 3–5
assumption questions that may semantically overlap with the shared assumptions
already collected, causing the user to answer the same question twice in
different phrasing.

This feature introduces a typed `SharedAssumptions` dataclass that centralizes
parsing and resolution in one place, and a question dedup catalog that prevents
downstream agents from re-asking questions already answered.

## User Stories

### US-001-001: Consistent Assumption Resolution

**As a** solution architect reviewing the ROI dashboard
**I want** all three agents to use identical values for current annual spend,
labor rate, and user counts
**So that** the cost estimate, value drivers, and ROI calculation are
internally consistent.

**Acceptance Criteria:**
- GIVEN shared assumptions contain "annual_it_budget" = 2,000,000 WHEN the
  Cost agent reads `current_annual_spend` THEN it resolves to 2,000,000
- GIVEN the same shared assumptions WHEN the ROI agent reads
  `current_annual_spend` THEN it resolves to the same 2,000,000
- GIVEN shared assumptions contain ambiguous keys like "yearly_cost" and
  "total_annual_spend" WHEN `SharedAssumptions.from_dict()` parses them THEN
  the first matching value wins (deterministic, not random)

### US-001-002: No Duplicate Questions

**As a** user going through the assumption-gathering flow
**I want** the system to skip questions whose answers are already known from
shared assumptions
**So that** I don't answer the same question twice with potentially different
phrasing.

**Acceptance Criteria:**
- GIVEN shared assumptions already contain `current_annual_spend` WHEN the
  Cost agent generates usage assumptions THEN no question about annual/monthly
  IT spend appears
- GIVEN shared assumptions already contain `total_users` WHEN the BV agent
  generates questions THEN no question about headcount/total employees appears
- GIVEN an LLM-generated question with ID "monthly_it_spend" WHEN the dedup
  filter runs THEN the question is blocked (semantic alias of
  `current_annual_spend`)

## Functional Requirements

### FR-001-001: SharedAssumptions Dataclass

Introduce a `SharedAssumptions` dataclass in `state.py` with typed fields:
`current_annual_spend`, `hourly_labor_rate`, `total_users`,
`concurrent_users`, `data_volume_gb`, `timeline_months`,
`monthly_revenue`. Includes a `raw` dict preserving the original
LLM-generated keys.

- Input: raw `dict[str, Any]` from `state.shared_assumptions`
- Processing: `from_dict()` classmethod iterates keys, applies keyword matching
  to classify each value into a typed field. First match wins per field.
  Numeric parsing with `float()`, non-numeric values skipped. Keys starting
  with `_` are skipped.
- Output: `SharedAssumptions` instance with typed fields + raw dict
- Error handling: Non-parseable values silently skipped (logged at DEBUG)

### FR-001-002: AgentState.sa Property

Add a `@property` on `AgentState` returning `SharedAssumptions.from_dict(self.shared_assumptions)`.

- Input: none (reads `self.shared_assumptions`)
- Processing: delegates to `SharedAssumptions.from_dict()`
- Output: `SharedAssumptions` instance
- Error handling: if `shared_assumptions` is empty, returns default instance
  with all fields `None`

### FR-001-003: Remove Legacy Fuzzy Matchers

Delete `_CURRENT_SPEND_KEYS`, `_LABOR_RATE_KEYS`, and `_resolve_sa()` from
`roi_agent.py`. Delete equivalent fuzzy key matching logic from
`business_value_agent.py` and `cost_agent.py`. Replace all call sites across
all agents with `state.sa.<field>`.

- Input: N/A (code deletion)
- Processing: grep for `_resolve_sa`, `_CURRENT_SPEND_KEYS`,
  `_LABOR_RATE_KEYS`, and any ad-hoc keyword matching against
  `shared_assumptions` across all agents; replace with `state.sa.*`
- Output: zero references to legacy matchers in any agent
- Error handling: N/A

### FR-001-004: Assumption Question Dedup Catalog

Create `agents/assumption_catalog.py` with:
- `SHARED_ASSUMPTION_IDS`: set of canonical field names
- `SEMANTIC_OVERLAPS`: set of known LLM-generated aliases
- `BLOCKED_IDS`: union of both sets
- `filter_already_answered(questions, state)`: removes questions whose ID is
  in `BLOCKED_IDS` or already present in `state.shared_assumptions`

- Input: list of question dicts (each has `"id"` key), `AgentState`
- Processing: build blocked set = `state.shared_assumptions.keys()` ∪
  `BLOCKED_IDS`. Filter out questions whose `id` is in the blocked set.
- Output: filtered question list
- Error handling: if `questions` is empty, return empty list

### FR-001-005: Wire Dedup into Agents

Call `filter_already_answered()` in:
- `CostAgent.generate_usage_assumptions()` — after LLM generates questions
- `BusinessValueAgent.generate_assumptions()` — after LLM generates questions

- Input: LLM-generated question list
- Processing: apply `filter_already_answered`
- Output: reduced question list presented to user
- Error handling: if all questions filtered out, proceed with zero agent-specific
  questions (shared assumptions are sufficient)

## Non-Functional Requirements

### NFR-001-001: Backward Compatibility

Existing `state.shared_assumptions` dict format must remain unchanged.
`SharedAssumptions` is a read-only view — it never writes back to the dict.
Existing workflows that populate `shared_assumptions` continue to work.

### NFR-001-002: Deterministic Resolution

Given identical input dict, `SharedAssumptions.from_dict()` must always
produce identical typed fields. No randomness, no set ordering dependency.
Python 3.7+ guarantees dict insertion order — `from_dict()` iterates keys
in insertion order, so "first match wins" is deterministic for a given
input dict.

### Architecture Pattern

The orchestrator collects 4 baseline questions (users, spend, labor rate,
concurrent users) and stores raw LLM-generated key/value pairs in
`state.shared_assumptions`. Each agent then independently re-interprets these
keys via fuzzy matching against its own private key lists.

### Known Limitations

- `shared_assumptions` is `dict[str, Any]` — no schema, no validation
- Keys are LLM-generated strings — e.g. "annual_it_budget" vs "current_annual_spend"
- `_resolve_sa()` in roi_agent.py has a 3-level fuzzy fallback that can match wrong keys
- Cost agent never reads `current_annual_spend` at all (see FRD-004)
- BV agent reads it but via its own separate key matching logic
